# Contributing

Thanks for helping improve Awesome Scientific Peer Review.

## Before Submitting

- Check the README for duplicates and closely overlapping resources.
- Confirm that the resource directly supports scientific peer review, editorial work, rebuttal, revision, evaluation, or an essential building block.
- Prefer the original paper, official documentation, dataset page, model card, or maintained repository.
- Verify that every link works and is publicly accessible.
- Read the resource closely enough to explain why it is useful.
- Do not submit confidential, private, leaked, or unethically collected review data.
- Disclose in the pull request if you are an author, maintainer, employee, or otherwise affiliated with the resource.

## Quality Criteria

A resource should be:

- Relevant to the scope of scientific peer review rather than software code review.
- Genuinely useful to researchers, reviewers, editors, students, or tool builders.
- Supported by a reliable primary source.
- Documented well enough for readers to understand or use it.
- Described accurately without marketing language or inflated capability claims.
- Maintained, historically important, or uniquely valuable.

Commercial and proprietary tools are welcome when they fill a clear editorial need, but their proprietary status must be stated.

## Adding an Entry

1. Choose the most specific existing section.
2. Add the entry in alphabetical order unless the section uses an intentional conceptual order.
3. Use this format:

   ```markdown
   - [Resource Name](https://original-source.example) - One concise sentence explaining what it is and why it is useful.
   ```

4. Put compact secondary links after the description:

   ```markdown
   - [Paper Title](https://paper.example) - Concise description. ([Code](https://code.example), [Data](https://data.example))
   ```

5. Start the description with an uppercase letter and end it with a period.
6. Keep the entry concise, grammatical, and free of unsupported claims.

Do not add a resource to several sections. Place it where it is most useful and mention its additional roles in the description.

## Pull Requests

- Keep each pull request focused.
- Explain why the resource belongs on the list.
- Include evidence that you checked the original source and link.
- Update related wording if a project was renamed, archived, superseded, or substantially changed.
- Be ready to revise or remove an entry if maintainers or contributors identify quality, licensing, privacy, or relevance concerns.
